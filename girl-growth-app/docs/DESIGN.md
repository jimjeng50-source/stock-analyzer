# 美少女養成陪伴 App — 設計文件 v1

> 現代都市背景的互動式養成陪伴 Web App。玩家扮演養育者，陪伴一位女兒從童年成長到成年；透過安排行程、日常對話與教養選擇，塑造她的能力、性格與人生，最終走向多種可能的結局。靈感來源：《美少女夢工廠》(Princess Maker)。

本文件為概念與規格的正式紀錄，作為後續開發的依據。內容經與需求方逐項確認而成。

---

## 1. 產品定位

| 項目 | 內容 |
|------|------|
| 產品類型 | 互動式養成陪伴 Web App |
| 核心樂趣 | **劇情與情感陪伴**（陪伴女兒長大的過程與情感連結） |
| 世界觀 | **現代都市**（貼近現實的生活、學業、才藝、職涯） |
| 語言 | **繁體中文**（介面、劇情、AI 對話） |
| 平台 | **網頁 Web App**（桌面 / 行動瀏覽器皆可） |
| 調性 | 成長養育為主、清新健康；女兒**成年後**可有健康的戀愛/情感結局 |

### 內容底線（硬性規範）
- **絕對禁止**任何性化、裸露、情色取向內容，不論角色年齡設定為何。
- 角色被設定為「養育對象／女兒」且養成自童年開始，此類角色的任何性化內容**一律不製作**。
- 「成人導向畫風」的定義**僅限於**：成熟精緻的美術風格 + 成年角色的時尚/優雅造型（如禮服、職業裝、優雅服飾），**不涉及情色**。
- AI 生成內容（文字與圖像）皆須套用上述界線作為系統層級的守則（guardrail）。

## 2. 核心玩法

### 2.1 玩法循環（Core Loop）
```
        ┌─────────────────────────────────────────────┐
        │                  一個回合                     │
        │  (時間單位：月 / 季，從童年推進到成年)          │
        └─────────────────────────────────────────────┘
                              │
        ┌─────────────────────┼─────────────────────┐
        ▼                     ▼                     ▼
   ① 安排行程/課程        ② 對話 / 教養選擇     ③ 隨機事件判定
   (培養能力數值)        (塑造性格/親密度)      (機率觸發)
        │                     │                     │
        └─────────────────────┼─────────────────────┘
                              ▼
                    ④ 結算：數值變化、狀態更新、劇情推進
                              ▼
                    ⑤ 達到成年 → 依條件觸發結局
```

- **回合制**：以「月」或「季」為時間單位（MVP 採**季**，一年 4 回合，降低回合數）。
- 每回合玩家做兩件主動決策：**安排行程**（培養能力）＋**對話/教養選擇**（塑造性格與親密度）。
- 每回合有機率觸發**隨機事件**。
- 回合結束後結算數值、更新狀態、推進主線劇情。
- 抵達成年（結局年齡）時，依累積的數值 / 性格 / 親密度 / 教育方向，判定並演出結局。

### 2.2 年齡與回合設定
- **完整版**目標：童年（約 10 歲）→ 成年（18 歲），以季為單位約 32 回合。
- **MVP**：壓縮為一小段（見 §9），僅需數回合即可跑完並觸發結局，用於驗證核心體驗。

## 3. 數值系統（四大面向）

女兒以四個面向描述，共同決定劇情走向與結局。

### 3.1 ① 能力數值（她「會什麼、多強」）
反映技能與才能，直接影響可達成的職業/人生結局。MVP 建議欄位：

| 數值 | 說明 |
|------|------|
| 體力 (Physique) | 運動、體育相關結局 |
| 智力 (Intellect) | 學業、研究、專業相關結局 |
| 才藝 (Talent) | 藝術、音樂、表演相關結局 |
| 社交 (Social) | 人際、領導、商業相關結局 |

> 完整版可再細分（如：學識/藝術/運動/口才/魅力…），MVP 先用上述 4 項。

### 3.2 ② 親密度 / 情感（呼應「陪伴」主題）
- **親密度 (Affinity)**：女兒對養育者的信任與情感（0–100）。
- 影響：她是否聽從安排、對話反應、能否觸發情感結局、部分事件選項是否開放。
- 是本作情感核心，設計上讓「陪伴、對話、關心」能穩定累積親密度。

### 3.3 ③ 性格 / 價值觀（她「是什麼樣的人」）
以幾組對立軸或標籤表示，由對話、教養選擇、事件選擇長期累積塑造。MVP 建議：

| 軸線 | 一端 ↔ 另一端 |
|------|------|
| 善惡傾向 | 善良 ↔ 現實 |
| 外向性 | 內向 ↔ 外向 |
| 自我 | 順從 ↔ 獨立/叛逆 |

- 性格影響：對話語氣、事件走向、結局氛圍與可選結局類型。

### 3.4 ④ 壓力 / 健康狀態（養成的節制與真實感）
- **壓力 (Stress)**：行程排太滿、事件衝擊會上升；休息、正向互動會下降。過高會反效果（數值成長打折、觸發負面事件、親密度下降）。
- **健康 (Health)**：影響能否正常進行行程；生病事件會降低。
- 作用：迫使玩家在「培養」與「照顧」之間取得平衡，增加拖拉與真實感。

## 4. 行程系統（每回合的能力培養）

每回合玩家為女兒安排一項（或有限數量）行程，消耗時間、可能消耗金錢，換取數值成長與狀態變化。

| 行程類別 | 主要提升 | 副作用 |
|----------|----------|--------|
| 學習類（補習、讀書） | 智力 | 壓力↑ |
| 才藝類（音樂、繪畫、舞蹈） | 才藝 | 壓力↑、金錢↓ |
| 運動類（體育、社團） | 體力 | 壓力↓（適度）|
| 社交類（活動、聚會） | 社交、親密度 | 金錢↓ |
| 休息 / 陪伴 | 壓力↓、健康↑、親密度↑ | 無能力成長 |
| 打工（年齡足夠後） | 金錢↑、社交↑ | 壓力↑ |

- **金錢系統**：養育者有預算，部分行程需付費，打工可賺取。MVP 可簡化為單一「金錢」數值。
- 行程效果會受**壓力/健康**與**親密度**調節（壓力高→成長打折；親密度高→更願意配合）。

---

## 5. 對話 / 教養系統（性格與親密度塑造）

- 每回合提供一段與女兒的互動：她可能提出問題、分享心情、或面臨小抉擇。
- 玩家透過**選項**回應（教養態度：鼓勵/嚴格/放任/傾聽…）。
- 影響：性格軸線、親密度、壓力，並可能開啟後續劇情。

### 5.1 AI 的角色（劇情為主 + AI 點綴）
- **主線與關鍵事件**：預先撰寫的劇情與選項，確保品質與結局連貫性。
- **日常對話**：由 **Claude AI** 即時生成，讓女兒的閒聊/回應更自然、有沉浸感。
- AI 生成須遵守：
  - 帶入**角色設定卡**（當前年齡、性格數值、近期事件、親密度）作為 context。
  - 套用內容界線（§1 底線）作為 system prompt 守則。
  - 輸出結構化資料（對話文字 + 對數值的建議影響），由後端驗證後套用，避免 AI 隨意改動數值。

## 6. 隨機事件系統

每回合結算前，依機率與條件觸發事件。四種類型皆納入：

| 類型 | 說明 | 範例 |
|------|------|------|
| ① 日常生活小事件 | 增加真實感與陪伴互動 | 女兒生病、交到朋友、考試、情緒低落、想要某樣東西 |
| ② 抉擇式分歧事件 | 突發狀況要玩家做決定，選項影響數值/性格/親密度 | 女兒迷路、被欺負、想輟學、朋友邀約 |
| ③ 機遇 / 奇遇事件 | 隨機出現的機會，帶來罕見成長或劇情轉折 | 星探洽談、比賽邀請、獎學金、貴人相助 |
| ④ 季節 / 節日事件 | 隨時間觸發，增加儀式感與時間累積感 | 生日、過年、暑假、聖誕節 |

- 事件由**觸發條件**（年齡、數值、性格、機率、季節）決定何時出現。
- 事件內容以**預寫事件庫**為主（保證品質與判定明確）；日常小事件的敘述可由 AI 潤飾。
- 事件結果會回饋到四大數值面向。

---

## 7. 結局系統

- 抵達成年時，依 **教育方向（累積的能力偏重）＋ 數值門檻 ＋ 性格 ＋ 親密度** 判定結局。
- 結局分類（現代都市範例）：
  - **能力導向職業結局**：學者/研究員（智力）、藝術家/音樂家（才藝）、運動員（體力）、創業家/領導者（社交）…
  - **性格 / 生活結局**：依性格軸線給出不同人生樣貌與氛圍。
  - **情感 / 陪伴結局**：高親密度時開放，強調母女/親子情感（女兒成年後亦可有健康戀愛結局）。
- **判定原則**：先看是否達成特殊結局的高門檻條件，否則落入依主導數值決定的一般結局；親密度與性格作為結局「風味」與分支修飾。
- MVP：先實作 **1–2 個結局**（例如「學業成就結局」與「才藝成就結局」，或「高親密度陪伴結局」），驗證判定流程。

---

## 8. 視覺呈現（AI 即時生成圖像）

- 女兒的立繪 / 場景由 **AI 生圖服務**即時生成，可隨**年齡與情緒/劇情**變化。
- **畫風**：成熟精緻（寫實或高質感插畫感），成年後走**時尚/優雅造型**；**無情色內容**。
- **關鍵技術挑戰 — 長相一致性**：
  - 維護一份**固定角色設定**（外貌 prompt 描述：髮色、瞳色、臉部特徵、風格關鍵字），每次生圖都帶入。
  - 以固定 seed / 參考圖（若服務支援 image-to-image 或 reference）維持一致性。
  - 依年齡階段預生成/快取幾張基準立繪，減少即時生成成本與不一致風險。
  - MVP 可先以「少量預生成立繪 + 快取」驗證流程，再擴充即時生成。

## 9. MVP 範圍定義

**目標**：用最小可玩範圍驗證「核心好不好玩」，並真接 AI。

MVP 必含：
1. **一小段養成循環**：約 4–8 個回合（如童年一年到隔年）。
2. **行程系統**：至少 4–5 種行程，影響四大數值。
3. **對話系統**：每回合一段互動 + 選項，含 **Claude AI 即時生成**的日常對話。
4. **四大數值面向**：能力、親密度、性格、壓力/健康，皆可運作並顯示。
5. **隨機事件**：四類各至少 1–2 個範例事件，含觸發與結算。
6. **結局**：1–2 個結局，含判定與演出。
7. **AI 生圖**：至少女兒立繪隨情緒/年齡呈現（先預生成+快取亦可）。
8. **前後端分離架構**：後端持有並保護 API 金鑰。

MVP 暫不含：完整 32 回合、大量事件庫、完整結局圖鑑、存檔雲端同步、多角色（同學/朋友立繪）、音效音樂。

## 10. 技術架構

### 10.1 總覽（前後端分離）
```
┌────────────────┐     HTTPS      ┌─────────────────────┐
│  前端 (瀏覽器)  │ ─────────────▶ │   後端 API (本機)    │
│  React + Vite  │ ◀───────────── │  持有 API 金鑰       │
│  遊戲 UI/狀態   │   遊戲/AI 請求  │  代理 AI 呼叫、驗證   │
└────────────────┘                └──────────┬──────────┘
                                             │
                              ┌──────────────┼──────────────┐
                              ▼                             ▼
                     ┌─────────────────┐          ┌──────────────────┐
                     │ Claude API      │          │ 生圖服務 API      │
                     │ (對話文字)      │          │ (圖像生成)        │
                     └─────────────────┘          └──────────────────┘
```

### 10.2 建議技術選型（可調整）
| 層 | 選型 | 理由 |
|----|------|------|
| 前端 | React + Vite + TypeScript + Tailwind CSS | 開發快、生態成熟、型別安全 |
| 狀態管理 | Zustand（或 React Context） | 輕量，適合遊戲狀態 |
| 後端 | Node + Express + TypeScript（或 Python FastAPI） | 藏金鑰、代理 AI、驗證 AI 輸出；前後端同語言(TS)較省心 |
| AI — 對話 | Claude API（Anthropic SDK，模型：claude-opus-4-8 / 依需求選型） | 需求方指定，品質佳 |
| AI — 生圖 | 生圖服務 API（如 OpenAI 影像生成等，依金鑰而定） | 需求方持有金鑰 |
| 存檔 | MVP 用瀏覽器 localStorage（後端 session 亦可） | 先本機、預留上線 |

> **金鑰安全**：金鑰只放後端 `.env`，前端**不接觸**金鑰；所有 AI 呼叫經後端代理。此設計「先本機、預留上線」——之後部署到雲端時後端可直接沿用。

### 10.3 AI 呼叫的資料契約（重點）
- 後端向 AI 送出：角色設定卡（年齡、四大數值、近期事件）+ 情境 + **內容界線 system prompt**。
- AI 回傳：**結構化 JSON**（如 `{ "dialogue": "...", "mood": "happy", "suggestedEffects": {...} }`）。
- 後端**驗證並箝制**數值影響範圍後才套用，避免 AI 直接操控存檔或越界。

## 11. 分階段開發計畫

> 每階段結束都有可展示成果；先求跑得動，再求豐富。

### 階段 0：專案骨架（地基）
- 建立 `girl-growth-app/`：前端（React+Vite+TS+Tailwind）與後端（Express+TS）專案骨架。
- 後端 `.env` 讀取 API 金鑰；一支健康檢查與一支「AI 對話代理」測試端點。
- 前端能呼叫後端、顯示回應。**驗收**：前端按鈕 → 後端 → Claude 回一句話。

### 階段 1：核心資料模型與回合骨架
- 定義女兒狀態資料結構（四大數值面向）與遊戲狀態（年齡、回合、金錢）。
- 實作回合推進、數值顯示 UI（狀態面板）。
- 存檔到 localStorage。**驗收**：能空跑數個回合、看到年齡/季節推進。

### 階段 2：行程系統
- 實作 4–5 種行程、選擇 UI、結算數值（含壓力/健康調節）。
- **驗收**：安排行程 → 數值正確變化、壓力/金錢連動。

### 階段 3：對話 / 教養系統（接 AI）
- 每回合互動 + 選項；日常對話接 Claude（結構化輸出 + 後端驗證）。
- 選項影響性格/親密度/壓力。**驗收**：對話自然、選擇改變性格與親密度。

### 階段 4：隨機事件系統
- 事件資料結構、觸發條件引擎、四類各數個範例事件、結算。
- **驗收**：回合中會依機率/條件跳出事件，選擇有後果。

### 階段 5：結局系統
- 結局判定引擎 + 1–2 個結局演出畫面。
- **驗收**：跑到成年 → 依數值/親密度觸發對應結局。

### 階段 6：AI 生圖與視覺
- 角色設定卡 + 一致性策略；女兒立繪依情緒/年齡呈現（先預生成+快取）。
- **驗收**：畫面出現一致風格的女兒立繪，隨情緒/年齡變化。

### 階段 7：MVP 整合與打磨
- 串起完整循環、UI 打磨、平衡數值、基本錯誤處理。
- **驗收**：從頭到尾玩完一局 MVP，體驗流暢。

---

## 12. 風險與注意事項

| 風險 | 對策 |
|------|------|
| AI 生圖角色長相不一致 | 固定設定卡 + seed/參考圖 + 預生成快取；MVP 先降低即時生成比例 |
| AI 對話越界或亂改數值 | system prompt 守則 + 結構化輸出 + 後端驗證箝制 |
| AI 呼叫成本/延遲 | 快取、限制頻率、日常對話才用 AI（主線用預寫） |
| 內容界線 | 全程套用 §1 硬性規範於所有 AI 生成 |
| 數值平衡 | MVP 先能跑通，數值調校放階段 7 與後續迭代 |

- 生圖服務的實際選型與 prompt 模板。
